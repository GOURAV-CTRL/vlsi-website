Sequence Detector "101" — Overlapping 🔍

All About VLSI · Lecture Notes — Part 23: Mealy & Moore, Side by Side
Problem Statement

Design a detector for the bit pattern 101 on a serial input din, with overlapping detection allowed — meaning the last bit of one match can be reused as the first bit of the next match (e.g. input 10101 should detect 101 twice, at positions 2 and 4).

Moore Machine — Output Depends Only on State
State Diagram Logic
  • S0 — idle / no match so far
  • S1 — saw "1"
  • S2 — saw "10"
  • S3 — saw "101" → detected=1 this cycle

For overlapping detection, from S3 (just matched "101"), the last bit "1" is reused — so on the next input, S3 behaves like S1 (already has one "1" queued), not like resetting fully to S0.

Moore FSM — Verilog Code
module seq_detect_101_moore ( input wire clk, input wire rst_n, input wire din, output reg detected ); localparam S0 = 2'b00, // idle S1 = 2'b01, // saw "1" S2 = 2'b10, // saw "10" S3 = 2'b11; // saw "101" -> match reg [1:0] state, next_state; // Block 1: State register (sequential, non-blocking) always @(posedge clk or negedge rst_n) begin if (!rst_n) state <= S0; else state <= next_state; end // Block 2: Next-state logic (combinational, blocking) always @(*) begin case (state) S0: next_state = din ? S1 : S0; S1: next_state = din ? S1 : S2; S2: next_state = din ? S3 : S0; S3: next_state = din ? S1 : S2; // overlap: reuse last '1' as new S1 default: next_state = S0; endcase end // Block 3: Output logic (Moore -> depends only on state) always @(*) begin detected = (state == S3); end endmodule

Note the S3 → din=1 → S1 transition: instead of dropping to S0, the machine treats the reused "1" as the start of a fresh potential match — this is exactly what makes overlapping detection work.

Mealy Machine — Output Depends on State + Input
Key Difference from Moore

A Mealy machine needs only 3 states for this problem (no separate "match" state) since the detected output can fire immediately, combinationally, on the transition S2 --din=1--> back toward S1 — the output is asserted during the transition rather than needing its own state.

Mealy FSM — Verilog Code
module seq_detect_101_mealy ( input wire clk, input wire rst_n, input wire din, output reg detected ); localparam S0 = 2'b00, // idle S1 = 2'b01, // saw "1" S2 = 2'b10; // saw "10" reg [1:0] state, next_state; // Block 1: State register (sequential, non-blocking) always @(posedge clk or negedge rst_n) begin if (!rst_n) state <= S0; else state <= next_state; end // Block 2: Next-state logic (combinational, blocking) always @(*) begin case (state) S0: next_state = din ? S1 : S0; S1: next_state = din ? S1 : S2; S2: next_state = din ? S1 : S0; // overlap: reuse '1' as new S1 default: next_state = S0; endcase end // Block 3: Output logic (Mealy -> depends on state AND input) always @(*) begin detected = (state == S2) && din; // fires the instant '1' completes "101" end endmodule

The detected signal here is a function of state == S2 and din == 1 simultaneously — it asserts one cycle earlier relative to the clock than the Moore version's registered output, which is the classic Mealy-vs-Moore timing difference.

Feed both modules the identical bitstream 1 0 1 0 1 and compare detected waveforms — the Mealy output will assert one clock cycle earlier than the Moore output for the same detected pattern, even though both correctly flag two overlapping matches.
