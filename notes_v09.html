Intro to Behavioural Modelling 📒

All About VLSI · Lecture Notes
Three Ways to Describe Hardware in Verilog

Verilog lets you describe the same circuit in more than one way, depending on how close to the hardware you want to write:

  • Gate Level → describe the circuit as interconnected primitive gates (and, or, not...). Closest to the actual transistors/schematic.
  • Data Flow → describe the circuit as continuous assignments using boolean/arithmetic expressions (the assign statement).
  • Behavioural → describe what the circuit should do, using procedural code (always / initial) — closer to software.

★ All three styles can even be mixed together in the same module — this is called mixed-style modelling.

What is Behavioural Modelling?

Behavioural modelling describes a circuit's functionality / behaviour rather than its exact gate-level structure — you tell Verilog "what should happen", not "how it's wired."

  • Written using procedural blocks: always and initial.
  • Uses high-level constructs like if-else, case, for, while — just like software.
  • Signals assigned inside these blocks must be declared as reg (or reg-like types), not wire.
  • The synthesis tool figures out the actual gates needed to implement the described behaviour.
module mux2 ( input a, b, sel, output reg y ); always @(*) begin if (sel) y = b; else y = a; end endmodule
How It Differs — Same 2:1 MUX, 3 Styles
🔹 Gate Level

Every gate is manually placed and wired — you decide the exact structure.

wire nsel, t1, t2; not (nsel, sel); and (t1, a, nsel); and (t2, b, sel); or (y, t1, t2);
🔹 Data Flow

One continuous expression describes the logic — no gates named, but still purely combinational and boolean/arithmetic in nature.

assign y = sel ? b : a;
🔹 Behavioural

Described with an if-else inside always — reads like a decision, not a circuit.

always @(*) y = sel ? b : a;

★ All three produce the exact same hardware after synthesis — they're just different levels of abstraction for describing it.

Why Designers Prefer Behavioural Modelling
  • Faster to write and read for complex logic — no need to derive boolean equations by hand.
  • Naturally handles sequential circuits (flip-flops, counters, FSMs) using @(posedge clk) — something gate/data-flow struggle to express cleanly.
  • Scales better for large designs — writing thousands of gates by hand is impractical.
  • Still synthesizable as long as you follow proper RTL coding guidelines — it's not just for testbenches.
Important Warnings
⚠️ Behavioural ≠ Always Synthesizable

Not everything you can write inside an always block can be turned into real hardware — things like #delay statements or real numbers are simulation-only and get ignored/rejected by synthesis tools.

⚠️ reg Doesn't Mean Flip-Flop

A signal declared as reg inside behavioural code becomes a flip-flop only if it's assigned inside a clocked always block; if it's driven by a combinational always @(*), it stays combinational logic.

⚠️ Don't Assign the Same Signal in Two Styles

A signal driven by assign (data flow) can't also be assigned inside an always block (behavioural) — this causes a multiple-driver conflict at compile/elaboration time.
