Half Adder & Full Adder — Behavioral 🧮

All About VLSI · Lecture Notes — Part 6: Combinational Building Blocks
Half Adder — The Basics

Adds two single bits (a, b) and produces a sum and a carry — but cannot accept an incoming carry, which is exactly why it's called "half."

absumcarry
0000
0110
1010
1101

Boolean equations: sum = a ⊕ b, carry = a · b.

Half Adder — Behavioral Code

Behavioral modeling describes what the circuit should do, using an always block, instead of wiring individual gates (that would be structural/dataflow style).

module half_adder ( input wire a, b, output reg sum, carry ); always @(*) begin sum = a ^ b; // XOR gives sum carry = a & b; // AND gives carry end endmodule
🔹 Key Points
  • Outputs are declared reg because they're assigned inside an always block — this does not imply a physical register, it's purely a Verilog data-type rule.
  • always @(*) is a combinational block — it automatically re-evaluates whenever a or b changes.
  • Both assignments use blocking (=), correct for combinational logic per the golden rule from Part 3.
Full Adder — The Basics

Adds three bits — two operands (a, b) plus an incoming carry (cin) — producing sum and cout. This is the building block that lets adders be chained to form multi-bit ripple-carry adders.

00110

Boolean equations: sum = a ⊕ b ⊕ cin, cout = (a·b) + (b·cin) + (a·cin).

Full Adder — Method 1: Direct Boolean Equations
module full_adder ( input wire a, b, cin, output reg sum, cout ); always @(*) begin sum = a ^ b ^ cin; cout = (a & b) | (b & cin) | (a & cin); end endmodule

★ This is the most direct translation of the truth table's Boolean equations — clean and easy to verify against the table above.

Full Adder — Method 2: Using a Case Statement

Instead of writing equations, you can describe the full adder purely from its truth table using a case statement on the concatenated inputs — useful for visualizing the table directly in code.

module full_adder_case ( input wire a, b, cin, output reg sum, cout ); always @(*) begin case ({a, b, cin}) 3'b000: {cout, sum} = 2'b00; 3'b001: {cout, sum} = 2'b01; 3'b010: {cout, sum} = 2'b01; 3'b011: {cout, sum} = 2'b10; 3'b100: {cout, sum} = 2'b01; 3'b101: {cout, sum} = 2'b10; 3'b110: {cout, sum} = 2'b10; 3'b111: {cout, sum} = 2'b11; default: {cout, sum} = 2'bxx; endcase end endmodule
  • {a, b, cin} concatenates the three inputs into one 3-bit selector for the case statement.
  • {cout, sum} on the left assigns both outputs together in one shot per case.
  • A default branch is included to avoid an unintended latch — see the warning below.
Full Adder — Method 3: Built from Two Half Adders

Structurally, a full adder is famously built by chaining two half adders plus an OR gate — this version instantiates the half_adder module already written above, and layers a small always block on top for the final OR.

module full_adder_from_ha ( input wire a, b, cin, output wire sum, cout ); wire s1, c1, c2; half_adder HA1 (.a(a), .b(b), .sum(s1), .carry(c1)); half_adder HA2 (.a(s1), .b(cin), .sum(sum), .carry(c2)); assign cout = c1 | c2; endmodule

★ This is technically a structural/mixed style (module instantiation), included here because it's the classic way full adders are taught — the first two methods above are the pure behavioral versions.

Important Warnings
⚠️ Missing default in case → Unwanted Latch

Leaving out the default branch in the case-statement version means the synthesizer can't guarantee sum and cout are assigned for every possible input combination — this infers an unintended latch instead of pure combinational logic.

⚠️ reg Does Not Mean Register

Declaring sum and cout as reg inside a combinational always block does not create flip-flops — reg is just the data type required for anything assigned inside a procedural block; the actual hardware here is pure combinational logic.

⚠️ Sensitivity List Must Be Complete

Using always @(*) instead of manually listing a, b, cin avoids the classic bug of forgetting a signal in the sensitivity list, which would make the block simulate incorrectly even though the logic equations are right.

⚠️ Half Adder Cannot Be Chained Alone

Because a half adder has no cin input, chaining half adders directly (without the extra OR gate for combining carries) cannot correctly form a multi-bit adder — this is exactly why the full adder exists as the real chainable building block.
