The Family of Display Tasks

All four are system tasks used to print simulation output to the console/log — but they differ in when they print, how often they print, and whether they add a newline. Picking the wrong one is a very common source of confusing debug output.

TaskNewline?When It FiresFires Repeatedly?
$displayYesImmediately (Active region)No — one-shot
$writeNoImmediately (Active region)No — one-shot
$strobeYesEnd of time step (Postponed region)No — one-shot
$monitorYesEnd of time step (Postponed region)Yes — auto, on any listed signal change
$display — Print Right Now

Executes exactly where it appears in the code, in the Active region, just like a blocking assignment. Automatically appends a newline after printing.

initial begin a = 4'd5; $display("a = %d", a); // prints "a = 5" immediately end

★ If a non-blocking assignment updates a signal on the same line/timestep just before a $display, the printed value can be the old one — because $display doesn't wait for the NBA Update region.

$write — Print Without a Newline

Behaves identically to $display in terms of timing (Active region, one-shot) — the only difference is it does not add a trailing newline, so multiple calls print on the same line.

$write("a = %d", a); $write(", b = %d", b); // output on ONE line: a = 5, b = 3
  • Handy for building up a custom formatted row across several calls.
  • Remember to manually insert \n if you want a line break — it won't happen automatically.
$strobe — Print the Final, Settled Value

Same syntax as $display, but scheduled to execute in the Postponed region — meaning it waits until every other event in that time step has finished, including all pending non-blocking updates, before actually printing.

always @(posedge clk) begin q <= d; $display("display: q = %b", q); // OLD q (Active region) $strobe ("strobe : q = %b", q); // NEW q (Postponed region) end

★ Use $strobe whenever you want to confirm the truly final value of a signal after a non-blocking assignment in the same block — it's the safe, race-free way to log post-update state.

$monitor — Auto-Print on Every Change

Called once in the code, but keeps firing automatically for the rest of simulation, once per time step, whenever any signal in its argument list changes value. Also lives in the Postponed region, so it always shows settled values.

initial $monitor("time=%0t a=%b b=%b sum=%b", $time, a, b, sum); // no need to call this again — it "watches" a, b, sum // forever, printing one line whenever any of them changes
🔹 Key Behaviors
  • Only one active $monitor is allowed at a time — calling it again replaces the previous argument list, it doesn't stack.
  • $monitoron / $monitoroff can temporarily pause and resume it without losing the argument list.
  • If multiple signals in the list change in the same time step, it still prints just once for that step — not once per signal.
Choosing the Right One
GoalBest Task

Important Warnings
⚠️ $display Right After <= Is a Trap

Printing a signal with $display immediately after assigning it with <= in the same always block will show the pre-update value, confusing beginners into thinking the non-blocking assignment "didn't work."

⚠️ Calling $monitor Twice Doesn't Stack

A second $monitor call with a different argument list replaces the first — it does not run both in parallel. If you need to watch two separate signal groups, combine them into a single $monitor call.

⚠️ $write Needs Manual Newlines

Forgetting to add \n with $write causes output from unrelated parts of the testbench to run together on one unreadable line — easy to miss until the log gets messy.

⚠️ These Are Simulation-Only

All four tasks are non-synthesizable — they exist purely to help with simulation/debug and are stripped out (or flagged as unsynthesizable) by synthesis tools, so they can be used freely without affecting the final hardware.
