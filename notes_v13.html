The Verilog Event Scheduler ⏱️

All About VLSI · Lecture Notes — Part 4: Simulation Semantics
Why a Scheduler Even Exists

Real hardware is massively parallel — every gate and flip-flop updates at once. A simulator, though, runs on a single CPU thread, so it needs a disciplined way to decide what order to process events in at any given simulation time, without breaking the illusion of parallelism.

The event scheduler is that referee. Simulation time is split into discrete time steps, and each time step is further split into ordered regions — this is what makes blocking vs non-blocking behavior deterministic and repeatable.

The Four Main Regions (per time step)

Within one simulation time step, events are processed in this strict order — a region only starts once the one before it is completely empty:

🔹 What Happens in Each Region
  • Active — blocking assignments (=), continuous assign evaluation, and $display execute here, immediately and in program order.
  • Inactive — statements scheduled with an explicit #0 delay land here, processed right after the Active region empties.
  • NBA Update — this is where every <= right-hand side (already sampled back in the Active region) is finally written to its left-hand side, all together.
  • Postponed — read-only region for $strobe and $monitor, guaranteed to see final, settled values after everything else this time step is done.

★ If new events get added to an earlier region while a later one is running (e.g. a blocking assignment triggers another Active-region event), the simulator loops back — it never skips ahead until a region is truly empty.

Walking Through an Example
module dff2 (input clk, d, output reg q1, q2); always @(posedge clk) begin q1 <= d; q2 <= q1; end endmodule
🔹 At the Clock Edge
  • Active region: both RHS expressions (d and q1) are evaluated using their values from just before the edge — but the actual writes to q1/q2 are only scheduled, not yet performed.
  • NBA region: now q1 is written with the sampled d, and q2 is written with the old q1 — not the brand-new one — because that value was captured back in the Active region.

This is exactly why a 2-stage non-blocking shift register works correctly regardless of statement order — the scheduler, not the code order, guarantees the "read old, write new" semantics.

Why $monitor and $strobe Live in Postponed

$display prints immediately, in the Active region, so it can show stale values if it runs before an NBA update lands. $monitor and $strobe are deliberately delayed to the Postponed region so they always report the final, fully-settled value of every signal for that time step.

always @(posedge clk) begin q <= d; $display("q = %b", q); // prints OLD q (Active region) $strobe ("q = %b", q); // prints NEW q (Postponed region) end
Important Warnings
⚠️ $display Can Lie to You

Because $display executes in the Active region, printing a signal right after a non-blocking assignment to it will show the old value, not the one you just scheduled — use $strobe if you need the post-update value in the same statement.

⚠️ Race Conditions Between Blocks

Two separate always @(posedge clk) blocks both using blocking assignments to communicate with each other can produce simulator-order-dependent results — this is the classic textbook race condition that non-blocking assignments were introduced to eliminate.

⚠️ #0 Delays Are a Code Smell

Relying on #0 to "push" a statement into the Inactive region and dodge a race condition usually signals a deeper ordering bug in the design — real RTL should never depend on Inactive-region timing tricks to behave correctly.

⚠️ Synthesis Tools Don't Simulate

The event scheduler is a simulation-only concept — synthesis tools infer hardware structure directly from blocking/non-blocking usage patterns instead, which is exactly why mismatched styles (Part 3's golden rule) cause simulation-vs-synthesis mismatches rather than scheduler confusion in silicon.
