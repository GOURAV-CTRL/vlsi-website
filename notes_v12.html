Blocking vs Non-Blocking Assignments 📒

All About VLSI · Lecture Notes — Part 3: Procedural Assignments
Blocking Assignment ( = )

Executes immediately and sequentially — the statement must fully complete (evaluate RHS, update LHS) before the next line in the same always block runs. Behaves like ordinary C-style code.

always @(*) begin y1 = a & b; // step 1: finishes completely... y2 = y1 | c; // step 2: ...before this line even starts end
🔹 Where It's Used
  • Almost exclusively inside combinational always blocks (always @(*)).
  • Models instantaneous logic — like wires and gates settling with zero delay.
  • Order of statements matters a lot — later lines see the updated value of earlier ones.

★ If y1 is used before it's assigned in the block, it holds its previous (stale) value for that evaluation — a classic sign of incomplete sensitivity or wrong ordering.

Non-Blocking Assignment ( <= )

Schedules the update to happen at the end of the current time step — RHS values are all sampled first, using pre-update values, then every LHS updates together, in parallel.

always @(posedge clk) begin q1 <= d; // both RHS values sampled q2 <= q1; // using OLD q1, not the new one end // after clock edge: q2 gets the value q1 had // BEFORE this edge — true shift-register behavior
🔹 Where It's Used
  • Almost exclusively inside sequential always blocks (always @(posedge clk)).
  • Correctly models flip-flop behavior — every register updates simultaneously on the clock edge.
  • Order of statements doesn't matter — swapping q1/q2 lines above gives the identical result.

★ This is exactly how a 2-stage shift register is built — swap the assignment type to blocking and it silently breaks into a combinational chain instead.

Why the Difference Actually Exists

Verilog's event scheduler keeps separate regions per simulation time step. Non-blocking assignments don't skip the queue — they deliberately go into a later region so every register reads the "before" world and writes the "after" world without racing each other.

🔹 The Two Regions (Simplified)
  • Active region — blocking assignments execute here, immediately, in program order.
  • NBA (Non-Blocking Assign) update region — all <= right-hand sides were already sampled earlier; now the left-hand sides actually get written, all at once, after every always block has run.

★ This is exactly why a clocked always block full of <= gives the same result no matter what order the statements are written in — they all read the old state and write the new state as one atomic step.

Important Warnings
⚠️ Never Mix = and <= in the Same Always Block

Mixing blocking and non-blocking assignments to different signals in one always block is legal but strongly discouraged — it makes the execution order hard to reason about and is a common source of simulation-vs-synthesis mismatches.

⚠️ Using = in Sequential Logic Causes Race Conditions

Writing q1 = d; inside a posedge clk block instead of q1 <= d; can make downstream logic see the new value in the same clock edge instead of the old one — this silently turns a shift register into a combinational pass-through in simulation.

⚠️ Using <= in Combinational Logic Delays Updates

Non-blocking assignments in an always @(*) block still work in simulation but delay the visible update by one simulation delta cycle — this can cause extra simulation events and confuses tools/readers expecting instantaneous combinational behavior. Stick to blocking for combinational logic.

⚠️ The Golden Rule

Combinational always block → use =. Sequential (clocked) always block → use <=. Following this one rule avoids almost every blocking/non-blocking bug in RTL design.
