The Standard 3-Always-Block FSM Style

The most common, synthesis-friendly way to code an FSM splits it into three always blocks:

  • State register — clocked, non-blocking, holds current state.
  • Next-state logic — combinational, blocking, decides next state from current state + inputs.
  • Output logic — combinational (or clocked for Moore-registered outputs), decides outputs from current state (Moore) or current state + inputs (Mealy).
Example — Simple Moore FSM (Detects "10" sequence)
module fsm_detect10 ( input wire clk, input wire rst_n, input wire din, output reg detected ); // State encoding localparam S0 = 2'b00, // idle S1 = 2'b01, // saw '1' S2 = 2'b10; // saw '1','0' -> detected reg [1:0] state, next_state; // Block 1: State register (sequential, non-blocking) always @(posedge clk or negedge rst_n) begin if (!rst_n) state <= S0; else state <= next_state; end // Block 2: Next-state logic (combinational, blocking) always @(*) begin case (state) S0: next_state = din ? S1 : S0; S1: next_state = din ? S1 : S2; S2: next_state = din ? S1 : S0; default: next_state = S0; endcase end // Block 3: Output logic (combinational, Moore -> depends only on state) always @(*) begin detected = (state == S2); end endmodule

This split keeps each block's purpose obvious and matches the golden rule from Part 16: the clocked block uses <=, both combinational blocks use =.

Moore vs Mealy — Quick Note
TypeOutput depends onOutput block sensitivity
MooreCurrent state onlystate
MealyCurrent state + current inputsstate, inputs

Moore outputs change only on clock edges (via state), so they're glitch-free but may lag by a cycle. Mealy outputs can react within the same cycle as an input change, but are more prone to combinational glitches.

⚠️ Always Give Next-State Logic a default Case

The next_state combinational block must cover every possible state value (including unreachable/unused encodings) — usually via a default case — otherwise synthesis may infer an unintended latch instead of pure combinational logic.
