Decoder — The Basics

A decoder takes an n-bit binary code as input and activates exactly one of its 2ⁿ outputs — the opposite job of an encoder. It's essentially the "address decoding" building block used everywhere from memories to control logic.

in (2-bit)Active Output
00out0
01out1
10out2
11out3
2:4 Decoder — Behavioral, Using case
module decoder2to4 ( input wire [1:0] in, input wire en, output reg [3:0] out ); always @(*) begin out = 4'b0000; // default: all outputs off if (en) begin case (in) 2'b00: out = 4'b0001; 2'b01: out = 4'b0010; 2'b10: out = 4'b0100; 2'b11: out = 4'b1000; endcase end end endmodule
🔹 Key Points
  • The enable (en) signal is common in real decoders — when low, every output stays off regardless of in.
  • out is pre-assigned to all-zero before the case statement — same latch-avoidance pattern used for the Part 9 DEMUX.
  • This is functionally almost identical to a 1:4 DEMUX with a constant input tied to 1 — decoders and demuxes are close cousins.
2:4 Decoder — One-Line Version Using Shift

Since a decoder just needs a single 1 in the position given by in, the shift operator from the very first notes gives a compact one-liner.

module decoder2to4_shift ( input wire [1:0] in, input wire en, output wire [3:0] out ); assign out = en ? (4'b0001 << in) : 4'b0000; endmodule

★ 4'b0001 << in works because a logical left shift by in positions moves the single set bit exactly where it needs to go — no case statement required.

Encoder — The Basics

An encoder does the reverse job — it takes 2ⁿ input lines, of which exactly one is expected to be active, and produces the n-bit binary code identifying which input that was.

Active Inputout (2-bit)
in000
in101
in210
in311
4:2 Encoder — Behavioral, Using casez
module encoder4to2 ( input wire [3:0] in, output reg [1:0] out, output reg valid ); always @(*) begin valid = 1'b1; casez (in) 4'b???1: out = 2'b00; // in0 active (lowest priority match ignored here) 4'b??10: out = 2'b01; 4'b?100: out = 2'b10; 4'b1000: out = 2'b11; default: begin out = 2'b00; valid = 1'b0; // no valid one-hot input end endcase end endmodule
🔹 Key Points
  • casez with ? wildcards elegantly matches "this bit is 1, don't care about the rest" patterns.
  • A plain (non-priority) encoder technically only works correctly when input is one-hot — exactly one bit set; this is why the code above is really behaving like a simple priority encoder as soon as more than one bit is set.
  • valid flags whether a legitimate one-hot pattern was seen, useful for catching a malformed input in simulation.
Priority Encoder — Recap from Part 8

A true priority encoder relaxes the one-hot requirement — if multiple inputs are active, it outputs the code for the highest-priority one (usually the MSB). This was already built with a for loop in Part 8's notes; the casez version above is the same idea using a different construct.

// Recap: MSB-first priority using a for loop (see Part 8) for (i = 7; i >= 0; i = i - 1) if (req[i] && !valid) begin grant_idx = i[2:0]; valid = 1'b1; end

Important Warnings
⚠️ Plain Encoders Break on Multiple Active Inputs

A basic 4:2 encoder only produces a meaningful code when the input is truly one-hot — if two or more bits are set simultaneously, the output is ambiguous unless you deliberately design it as a priority encoder instead.

⚠️ Decoder Enable Must Gate All Outputs

Forgetting to gate every output bit with en means the decoder can still show a "decoded" output even when disabled — always pre-assign all outputs to 0 and only activate the selected one inside the enabled branch.

⚠️ casez Wildcards Must Not Overlap Unexpectedly

Because 4'b???1 also technically matches 4'b0011 or 4'b1111, the branch order in a casez-based encoder matters — the first matching pattern wins, so priority must be written from highest to lowest exactly as intended.

⚠️ valid Flag Should Not Be Skipped

Omitting a valid output on an encoder makes it impossible for downstream logic to distinguish "input code 00 because in0 was active" from "input code 00 because nothing was active" — always expose this distinction explicitly.
