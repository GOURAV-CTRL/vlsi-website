Why = for Combinational, <= for Sequential 🤔

All About VLSI · Lecture Notes — Part 16: The Golden Rule, Explained
The Rule, Quickly Recapped

Part 3 introduced this as the "golden rule": use blocking (=) in combinational always blocks, and non-blocking (<=) in clocked/sequential always blocks. This note explains why that rule exists, instead of just stating it.

Reason 1 — Each Style Models Different Real Hardware

Combinational logic (gates, muxes, adders) has no memory — its output is a pure function of its current inputs, updating "instantly" (well, after gate delay) whenever any input changes. Blocking assignment mirrors this: each statement finishes completely before the next runs, exactly like signals propagating through a chain of gates.

Sequential logic (flip-flops) is fundamentally different — every flop samples its input at the same instant (the clock edge) and holds it until the next edge. Non-blocking assignment mirrors this: every right-hand side is sampled from the "before-edge" world, and every left-hand side updates together, as if all flip-flops in the circuit are edge-triggered simultaneously — because in real hardware, they are.

Reason 2 — The Classic Shift Register Proof

This is the clearest demonstration. Recall the 2-flop chain from Part 4:

// Non-blocking version — CORRECT for flip-flops always @(posedge clk) begin q1 <= d; q2 <= q1; // uses OLD q1 (from before this edge) end
// Blocking version — WRONG for flip-flops always @(posedge clk) begin q1 = d; q2 = q1; // uses NEW q1 (just updated above!) end

With <=, q2 correctly gets the value q1 held before the clock edge — true 2-stage shift-register behavior, matching two real flip-flops in series. With =, q2 instantly gets q1's brand-new value in the same edge — collapsing two flops' worth of delay into one, which is not what the hardware actually does.

Reason 3 — It's Baked Into the Event Scheduler

Part 4 covered this in depth: blocking assignments execute in the Active region, immediately, in program order. Non-blocking right-hand sides are sampled in Active, but their left-hand sides are only written in the later NBA Update region — after every always block in that time step has been evaluated.

This scheduler design isn't arbitrary — it's precisely engineered to give sequential logic its "read-old, write-new, all-at-once" behavior, and to give combinational logic its "immediate, in-order" behavior.

Reason 4 — Non-Blocking Eliminates Race Conditions Between Blocks

Imagine two separate always @(posedge clk) blocks, each reading a signal the other writes. With blocking assignments, the result depends on which block the simulator happens to execute first in that time step — a genuine race condition, since Verilog doesn't guarantee execution order between separate always blocks.

// Block A always @(posedge clk) a = b; // race! depends on execution order // Block B always @(posedge clk) b = a;

With non-blocking, both blocks sample the pre-edge values of a and b in the Active region, then write in the NBA region — the result is deterministic regardless of which always block the simulator happens to visit first.

Reason 5 — Keeps Simulation and Synthesis in Agreement

Synthesis tools infer hardware structure from coding style — blocking assignments in a sensitivity-complete always block infer combinational gates; non-blocking assignments in a clocked always block infer flip-flops. Following the golden rule means the mental model you get from simulation (waveforms) matches the actual gate/flop-level circuit that gets built — mixing styles risks a mismatch where simulation "looks right" but synthesized hardware behaves differently, or vice versa.

Important Warnings
⚠️ "It Simulates Fine" Isn't Proof It's Correct

A design using blocking assignments in sequential logic can sometimes appear to simulate correctly in simple testbenches, purely by luck of statement/block ordering — this is exactly the kind of race condition that only shows up when block order changes or under different tool versions.

⚠️ The Rule Is About the Block Type, Not the Signal Name

The golden rule depends on whether the always block is combinational (@*) or clocked (@posedge clk) — not on any property of the signal itself. The same signal name could theoretically appear in either kind of block in different modules, and the assignment style must match the block it's actually written in.

⚠️ Breaking the Rule Doesn't Always Error Out

Verilog will happily compile and simulate code that breaks the golden rule — there's no built-in error for using = in a clocked block or <= in a combinational one. The consequences (races, synthesis mismatch) are silent until they cause a real functional bug, which is exactly why this rule is worth internalizing rather than relying on tools to catch it.
