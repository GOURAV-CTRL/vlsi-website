For Loops & Case Statements 🔁

All About VLSI · Lecture Notes — Part 8: Procedural Constructs
The for Loop — Syntax

Just like C, a for loop repeats a statement (or block) a fixed number of times, controlled by an initialization, a condition, and a step — but here it must live inside a procedural block (initial/always), never on its own.

integer i; always @(*) begin for (i = 0; i < 8; i = i + 1) begin // body executes once per iteration end end
🔹 Key Points
  • The loop variable (i here) is usually declared as integer — a 32-bit signed type — not as a wire/reg vector.
  • Unlike software, a Verilog for loop inside combinational logic is unrolled at compile/elaboration time — it doesn't create real hardware "looping," it just repeats the described logic N times.
  • Always wrap multi-statement bodies in begin ... end, exactly like any other procedural block.
Example 1 — Parity Generator (Reduction via Loop)

A for loop can XOR together every bit of a bus one at a time — useful before you learn the shorthand reduction operator (^data), and still handy for more complex per-bit processing.

module parity_gen ( input wire [7:0] data, output reg parity ); integer i; always @(*) begin parity = 1'b0; for (i = 0; i < 8; i = i + 1) begin parity = parity ^ data[i]; // accumulate XOR of all bits end end endmodule

★ Notice parity is reset to 0 before the loop starts — forgetting this initialization is one of the most common for-loop bugs, since it would otherwise hold a stale value from the previous evaluation.

Example 2 — Priority Encoder Using a Loop
module priority_enc8 ( input wire [7:0] req, output reg [2:0] grant_idx, output reg valid ); integer i; always @(*) begin grant_idx = 3'd0; valid = 1'b0; for (i = 7; i >= 0; i = i - 1) begin if (req[i] && !valid) begin grant_idx = i[2:0]; valid = 1'b1; end end end endmodule

Looping from MSB down to LSB and using a valid flag to "lock in" the first match found is a classic pattern for building priority logic without a long if-else chain.

The case Statement — Syntax

Selects one branch to execute based on matching a control expression against a list of possible values — much cleaner than a long chain of if-else-if for multi-way decisions.

reg [1:0] sel; reg [7:0] y; always @(*) begin case (sel) 2'b00: y = a; 2'b01: y = b; 2'b10: y = c; default: y = d; endcase end
🔹 Key Points
  • Each branch is checked top to bottom; the first match wins and the rest are skipped.
  • A default branch should almost always be included — see the warning below on latch inference.
  • Multiple values can share one branch: 2'b01, 2'b10: y = x; using a comma-separated list.
casez and casex — Wildcard Matching
StatementTreats as "don't care"
caseNothing — exact match only, including x/z
casezz bits (written as ?) in the case item
casexBoth x and z bits in the case item
always @(*) begin casez (opcode) 4'b1???: instr_type = ADD; // any value with MSB=1 matches 4'b01??: instr_type = SUB; default: instr_type = NOP; endcase end

★ casex is generally discouraged in synthesizable RTL — treating actual unknown x values in simulation as "don't care" can mask real bugs that casez would not hide.

Putting Them Together — BCD to 7-Segment Decoder

A great real example that uses a case statement to map each digit, entirely inside an always block:

module bcd_to_7seg ( input wire [3:0] digit, output reg [6:0] seg // {g,f,e,d,c,b,a} ); always @(*) begin case (digit) 4'd0: seg = 7'b0111111; 4'd1: seg = 7'b0000110; 4'd2: seg = 7'b1011011; 4'd3: seg = 7'b1001111; 4'd4: seg = 7'b1100110; 4'd5: seg = 7'b1101101; 4'd6: seg = 7'b1111101; 4'd7: seg = 7'b0000111; 4'd8: seg = 7'b1111111; 4'd9: seg = 7'b1101111; default: seg = 7'b0000000; // blank for invalid BCD endcase end endmodule
Important Warnings
⚠️ Missing default → Inferred Latch

Just like Part 6's full adder example, a case statement without a default branch (or that doesn't cover every possible value) leaves outputs unassigned for some inputs, causing the synthesizer to infer an unwanted latch instead of combinational logic.

⚠️ Loop Variable Must Be Reset Each Time

Any accumulator variable used inside a for loop (like the running XOR in the parity example) must be explicitly initialized before the loop on every invocation of the always block — otherwise it silently carries over its previous value.

⚠️ for Loops Don't Create Real Hardware Loops

A synthesizable for loop is fully unrolled into repeated combinational logic at synthesis time — it is not a runtime loop like in software, so loop bounds must be fixed at compile time (no data-dependent bounds).

⚠️ casex Can Hide Real Bugs

Because casex treats both x and z as wildcards, an actual unintended unknown value in simulation can accidentally match a branch instead of flagging a bug — prefer casez (or plain case) in most RTL code.
