Port Connection Rules 📒

All About VLSI · Verilog Interfacing Notes
A guide to correctly interfacing between modules and testbenches — how ports are typed, how connections are made, and the rules/pitfalls every beginner should know before VLSI / FPGA interviews.
✅ Port Data Types
🔹 Input Ports
  • Always of type net (wire) — never declared as reg inside the module.
  • Declaring an input as reg causes a compilation error, because inputs must be continuously driven from outside.
🔹 Output Ports
  • Can be declared as either net (wire) or reg.
  • Use reg if the output value needs to be stored/held inside a procedural block (e.g. assigned inside always).
  • Use wire if it's just a continuous combinational output (e.g. via assign).
🔹 Testbench ↔ Design Connections
  • Signals that drive the design's inputs from the testbench → can be reg or net.
  • Signals that receive the design's outputs back into the testbench → must be net (wire) only.

✅ Methods of Port Connection
🔹 Connection by Name (Named Association)
  • Uses the . syntax → .port_name(signal)
  • Recommended practice — readable and order-independent (won't break if port order changes in the module).
and_gate u1 ( .a(sig_a), .b(sig_b), .y(sig_y) );
🔹 Connection by Order (Positional Association)
  • Connects signals purely based on the order they appear in the module's port list.
  • Error-prone — risky as designs grow larger or port order changes.
and_gate u1 (sig_a, sig_b, sig_y);

★ Rule of thumb → always prefer named association in real projects; positional is okay only for very small/simple modules.

⚠️ Common Pitfalls & Rules
⚡ Width Mismatch

If the port width and connected signal width differ, Verilog only connects the LSBs — extra bits are silently dropped/truncated. No error is thrown, so this is considered poor practice and should be avoided.

⚡ Unconnected Inputs

Leaving an input port unconnected during instantiation triggers a compilation error — inputs must always have a driving source.

⚡ Unconnected Outputs

Leaving an output port unconnected is allowed and will not cause an error — useful when you don't need to observe that particular output.

⚡ Argument Mismatch

Providing the wrong number of arguments during instantiation (too many/too few) leads to a compilation error — the connection list must match the module's port list.

✅ Extra Tips — Good Practice Checklist
  • Always use named port connections for designs with more than a handful of ports.
  • Match port widths exactly — don't rely on implicit LSB truncation.
  • Double-check that every input is driven before instantiation — simulators will stop you, but it's good to catch early.
  • It's fine to leave unused outputs unconnected — use an underscore .port_name() or simply omit it when using named association.
  • When connecting from a testbench, remember: DUT inputs ← reg/wire, DUT outputs → wire only.
